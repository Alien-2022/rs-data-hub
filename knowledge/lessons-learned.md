### 坑 1: web_fetch 访问失败误判为链接失效

**问题描述**:
使用 web_fetch 工具验证 DIOR、MLRSNet 等数据集的论文链接时，返回访问失败，初步判断为链接失效。

**原因分析**:
- ScienceDirect 有反爬虫机制（机器人验证）
- arxiv 页面需要 JavaScript 渲染
- web_fetch 的 readability 提取器可能无法解析某些页面结构

**实际状况**:
用浏览器手动访问这些"失效"链接，发现全部正常！

**解决方案**:
1. **优先使用浏览器验证** - 最可靠
2. **使用 curl 检查 HTTP 状态码** - 快速判断
3. **web_fetch 失败不直接判断失效** - 需二次验证
4. **找替代平台** - 如 ScienceDirect 失败找 arxiv

**经验总结**:
```
❌ 错误做法：web_fetch 失败 → 标记为失效
✅ 正确做法：web_fetch 失败 → 浏览器验证 → 找替代平台 → 最终判断
```

**预防措施**:
- 在验证流程中明确：web_fetch 仅作为辅助工具
- 重要链接必须浏览器手动验证
- 记录验证工具和方法

### 坑 2: 过早判断链接失效

**问题描述**:
初步验证时，将 10 个数据集标记为"待验证"，实际这些链接都有效。

**原因分析**:
- 过度依赖自动化工具
- 没有手动验证
- 对付费墙论文直接判断为"不可访问"

**解决方案**:
1. **arxiv 优先策略** - 付费论文先找 arxiv 预印本
2. **期刊官网作为引导** - 不要求能直接下载 PDF
3. **多平台确认** - 一个平台失效找其他平台

**经验总结**:
```
验证优先级：
1. arxiv（开源稳定）⭐⭐⭐⭐⭐
2. 期刊官网（引导页面）⭐⭐⭐⭐
3. IEEE/ScienceDirect（付费墙）⭐⭐⭐
```

### 坑 3: GitHub 仓库 404 误判

**问题描述**:
SSDD 数据集的旧 GitHub 链接 404，初步判断为数据集不可用。

**原因分析**:
- GitHub 仓库可能迁移或改名
- 作者可能开新仓库
- 没有搜索作者新仓库

**解决方案**:
1. **搜索作者账号** - 找同名或相似账号
2. **搜索数据集名称** - 找镜像仓库
3. **检查仓库重定向** - GitHub 有时会保留重定向

**实际解决**:
```
旧链接：TianwenZhang0825/SSDD_Ship_Detection (404)
新链接：TianwenZhang0825/Official-SSDD (✅ 正常)
```

**经验总结**:
```
GitHub 404 ≠ 数据集消失
→ 搜作者新仓库
→ 搜数据集镜像
→ 联系作者
```

### 坑 4: 文档结构混乱

**问题描述**:
初期没有规划好文档结构，项目文档和经验总结混在一起。

**影响**:
- 每次加载大量不必要的内容
- 浪费上下文
- 难以查找特定信息

**解决方案**:
创建分层文档结构：
```
docs/           # 项目文档（稳定）
  ├── architecture.md
  ├── workflows/
  └── ...

knowledge/      # 经验总结（可进化）
  ├── verification-experience.md
  ├── lessons-learned.md
  └── ...

memory/         # 每日记录（临时）
  └── 2026-03-27.md
```

**经验总结**:
```
硬知识（项目架构） → docs/
软知识（经验技巧） → knowledge/
临时记录（任务进度） → memory/
```

### 坑 5: 没有及时记录验证经验

**问题描述**:
验证前 10 个数据集时走的弯路，在验证后 10 个时又重复了。

**原因分析**:
- 没有及时记录验证技巧
- 没有形成检查清单
- 经验没有沉淀

**解决方案**:
1. **边验证边记录** - 发现技巧立即记下
2. **创建检查清单** - 避免重复错误
3. **任务后总结** - 整理到知识库

**实际产出**:
- `knowledge/verification-experience.md` - 验证经验
- `docs/workflows/data-verification.md` - 验证流程
- `knowledge/lessons-learned.md` - 踩坑记录
